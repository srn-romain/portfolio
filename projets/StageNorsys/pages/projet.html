Projet CRM

Bienvenue sur le projet CRM, une expérience à laquelle j'ai eu le privilège de participer en tant que stagiaire chez Norsys. Ce projet a été développé par l'équipe de développement de l'entreprise, et en tant que stagiaire, j'ai pu y apporter ma contribution. Lancé en février 2023, ce projet est ambitieux et prévoit encore plusieurs années de développement avant sa finalisation. Toutefois, dès juillet 2023, il sera mis à disposition des commerciaux de l'entreprise Norsys, qui en seront les utilisateurs.

L'objectif principal du projet est de créer une interface rapide et intuitive pour les commerciaux, où chacun dispose d'un espace numérique personnalisé. Cet espace contient la liste des comptes clients de l'entreprise. Lorsqu'un commercial sélectionne un compte client, il peut accéder à une multitude d'informations, telles que les collaborateurs travaillant avec ce client (développeurs, chefs de projet, lead dev...), les contacts associés, les services liés, les compétences de chaque collaborateur et plus encore.

L'interface CRM du projet vise à faciliter le travail des commerciaux en leur fournissant toutes les informations nécessaires sur les comptes clients de manière claire et organisée. Cela leur permet de mieux comprendre les relations entre l'entreprise Norsys et ses clients, ainsi que de gérer efficacement leurs interactions avec eux. Ce CRM est donc une véritable opportunité de croissance pour l'entreprise.

Voici quelques captures d'écran de l'interface graphique de ce projet ambitieux développé en Angular et Java. Ces images illustrent la qualité et l'esthétique de l'interface utilisateur, fruit du travail minutieux de l'équipe de développement. Bonne visite !

Le sprint

Le projet fonctionne par sprint. Mais alors qu'est-ce que c'est ? Un sprint est une période de temps définie dans la méthodologie de développement Agile. Pendant un sprint, l'équipe de développement se concentre sur un ensemble spécifique de fonctionnalités ou de tâches à accomplir. Habituellement, un sprint dure entre une à quatre semaines. À la fin de chaque sprint, l'équipe de développement présente les fonctionnalités développées à Julien.V pour validation. Celui-ci est un commercial de l'entreprise qui est le client référent de ce projet. Cette approche itérative par sprint permet d'obtenir des retours réguliers du client tout au long du processus de développement et d'ajuster les fonctionnalités en fonction de ses besoins et de ses attentes.

Tests unitaires

Les tests unitaires sont une pratique de développement logiciel qui consiste à écrire des tests automatisés pour vérifier le bon fonctionnement des unités individuelles de code, telles que des classes, des méthodes ou des fonctions, dans une application. Ils nous aident à détecter les erreurs ou les régressions potentielles plus tôt dans le processus de développement, ce qui facilite la détection et la résolution des problèmes. Ils vérifient si chaque unité de code répond correctement aux différentes entrées et conditions attendues, en s'assurant que le code produit les résultats escomptés. Dans le cadre de notre projet CRM, les tests unitaires sont réalisés exclusivement sur la partie backend. Voici donc quatre raisons qui mettent en évidence l'importance des tests unitaires :

Détection précoce des erreurs
Facilité de maintenance
Régression minimale
Confiance dans le code


1. Deux approches

Il existe différentes approches et concepts liés aux tests unitaires, deux écoles diverge "London School" et la "Detroit School", les différentes issues des tests, la relation entre les tests unitaires et les tests d'intégration, ainsi que les piliers d'un bon test unitaire.

- La London School met l'accent sur l'utilisation de mocks (objets simulés) pour isoler les unités de code à tester.Cela permet de tester chaque unité indépendamment des autres, facilitant ainsi la détection des erreurs. Cette technique offre une isolation complète des unités de code qui permet de détecter les erreurs spécifiques à chaque unité
- La Detroit School, quant à elle, encourage l'utilisation de véritables dépendances lors des tests unitaires. Au lieu de se reposer sur des mocks, elle favorise l'intégration réelle des dépendances, ce qui permet de valider le fonctionnement du code dans un contexte plus proche de la réalité. Cette technique quant à elle offre une meilleure confiance dans le fonctionnement global du système en testant les véritables dépendances

2. Quatres issues des tests

Vrai positif :
le test signale une erreur lorsque le comportement attendu est incorrect.
Faux positif : le test signale une erreur alors que le comportement est correct.
Vrai négatif : le test ne signale aucune erreur lorsque le comportement est correct.
Faux négatif : le test ne signale pas d'erreur alors que le comportement est incorrect.

3. Tests unitaires vs tests d'intégration :


- Les tests unitaires se concentrent sur la validation de chaque unité de code de manière isolée.
- Les tests d'intégration évaluent le bon fonctionnement des différentes unités de code lorsqu'elles sont combinées.

Pyramide des tests :
- Les tests unitaires sont situés à la base de la pyramide des tests, représentant la majorité des tests effectués.
- Les tests d'intégration se trouvent au niveau intermédiaire, validant l'intégration entre les différentes unités.
- Les tests d'interface utilisateur se situent au sommet de la pyramide, s'assurant du bon fonctionnement global de l'application.

4. Les quatre piliers d'un bon test unitaire :


- Lisibilité : le test doit être facile à comprendre pour les développeurs et maintenable à long terme.
- Isolation : le test doit être indépendant des autres tests et ne pas dépendre d'un état externe.
- Rapidité : le test doit s'exécuter rapidement pour permettre une intégration continue et un retour rapide en cas d'erreur.
- Fidélité : le test doit être fiable et reproductible, donnant des résultats cohérents à chaque exécution.

En mettant en pratique ces concepts et en adoptant une approche appropriée, les tests unitaires deviennent un outil puissant pour garantir la qualité du code, faciliter la maintenance et améliorer la confiance dans le système développé.

Bienvenue dans la section "User Stories" de mon portfolio. Ici, je mets en avant le code des US pour lesquelles j'ai été responsable. Au cours de mon stage de 6 semaines, j'ai eu l'opportunité de gérer trois US. Dans les lignes qui suivent, je vais les détailler.

1. Architecture du projet

Tout d'abord, je vous présente trois captures d'écran qui illustrent l'architecture du projet CRM. La première capture dévoile l'architecture backend du projet, tandis que les deux captures suivantes mettent en évidence la partie frontend du projet.

2. Première US : le favicon

Pour commencer, en ce qui concerne ma première US, j'ai eu pour tâche d'ajouter un favicon. Cette modification a été effectuée du côté front, étant donné que le favicon est un élément visuel. Ensuite, dans le fichier principal 'index.html' du frontend, j'ai ajouter la ligne 8 encadrer en rouge.

3. Seconde US : rendre le texte cliquable

Ensuite, en ce qui concerne ma deuxième US, j'ai été chargé de rendre le texte du menu cliquable, car jusqu'alors seule l'icône avait cette fonctionnalité. C'est donc dans un premier temps dans la partie frontend que j'ai apporté des modifications. L'idée m'est venue de déplacer le contenu de la balise "span" (c'est-à-dire le texte que nous voulions rendre cliquable) à l'intérieur de la balise "a". Vous pouvez voir ci-dessous le résultat en image de la partie frontend. Le trait coloré représente son emplacement précédent.

Cependant, cela ne s'est pas avéré suffisant. Avec l'assistance de Guillaume, le spécialiste frontend du projet, j'ai appris qu'il était nécessaire de modifier le code SCSS de la balise "a" pour étendre la zone cliquable. C'est ainsi que j'ai pu coder le contenu des deux images suivantes, en bénéficiant d'un peu d'aide de Flavio et à nouveau de Guillaume. Au départ, la zone cliquable se résumait à un cercle, c'est-à-dire à l'icône. J'ai donc opéré un changement en transformant ce cercle en une forme de rectangle, qui rendrait cliquable son contenu. Certaines balises moins essentielles, telles que "&::after" (entouré en jaune), ont permis de paramétrer les effets lorsque la souris survole ce rectangle. Dans cet exemple, le texte modifie sa couleur et augmente légèrement en taille.

4. Troisième US : création de contact sans compte

Voila maintenant ma troisième et derniere US. Jusqu'à présent, lorsqu'il s'agissait de créer un nouveau contact, comme par exemple un nouveau développeur rejoignant l'entreprise, nous étions contraints de l'associer à un compte client existant. Cependant, si ce nouvel informaticien n'est pas encore affecté à un client spécifique, nous nous trouvions dans l'incapacité de créer ce contact, puisque l'attribution à un client était une obligation. Voici donc le cœur de ma troisième User Story : créer une page de création de contact vierge, sans l'associer à un compte client.
L'image de gauche illustre le code HTML du formulaire de création d'un contact vierge. Pour en arriver à cette étape, j'ai pu reprendre le formulaire de création de contact préexistant, à l'exception que cette fois-ci, nous ne lui associons pas de compte. En ce qui concerne l'image du centre et celle de droite, elles dévoilent les conditions à respecter (écrites en TypeScript) lors de la création d'un contact. Par exemple, aux lignes 81 à 85, nous pouvons constater que le nom doit obligatoirement être saisi pour que le contact soit validé (ce qui suggère que le nom est la clé primaire de la table). Diverses autres conditions sont également visibles dans la suite de l'image.

Lexique

Bienvenue dans la section "Lexique" de mon site, où vous trouverez le vocabulaire technique que j'ai utilisé au cours de mon stage. Il se peut que certains mots ne soient pas répertoriés dans le portfolio, mais ils ont été employés au cours de mon stage.


Peer-Programming

La peer programming (pair programming en francais), ou programmation en binôme, est une méthode de développement logiciel où deux programmeurs travaillent ensemble sur le même ordinateur, en partageant le clavier et l'écran, pour collaborer étroitement dans la résolution de problèmes, la conception et la rédaction de code.

Collaps

Collaps est un élément ou un composant d'interface utilisateur qui permet de réduire ou d'expandre du contenu de manière interactive, généralement à l'aide d'un bouton ou d'un lien, pour économiser de l'espace à l'écran.

Mob prog

Le "mob programming" est le terme utilisé pour décrire la pratique de plusieurs programmeurs travaillant ensemble sur un même projet ou un même ensemble de fonctionnalités.

Module

En Angular, le terme Module désigne un ensemble de fonctionnalités d'une application, il donne essentiellement des informations angulaires sur les fonctionnalités de celle-ci

Framework

Un framework est un ensemble d'outils, de bibliothèques et de conventions de développement préétablies qui facilitent la création et la gestion de logiciels. Il fournit une structure de base pour le développement d'applications en regroupant des fonctionnalités communes et en fournissant des abstractions pour simplifier les tâches récurrentes. Les frameworks permettent aux développeurs de gagner du temps en réutilisant du code existant et en suivant des bonnes pratiques établies. Ils offrent également une architecture cohérente et favorisent une approche modulaire et évolutive du développement logiciel. En résumé, un framework est un ensemble d'outils et de conventions qui accélèrent le processus de développement en fournissant une structure et des fonctionnalités prêtes à l'emploi.

Component

Un composant en Java Angular est une unité autonome de fonctionnalité qui combine du code, des modèles et des styles pour fournir une partie spécifique de l'interface utilisateur d'une application. Il encapsule la logique et les données liées à cette fonctionnalité, facilitant ainsi la modularité et la réutilisabilité du code.

M R

Les merge requests (MR), également appelées pull requests, sont une fonctionnalité essentielle dans les systèmes de contrôle de version tels que Git. Elles permettent aux développeurs de proposer des modifications de code à une branche principale ou à un référentiel central. Les merge requests permettent une collaboration efficace en facilitant la révision et la validation des changements par les pairs. Elles offrent également un historique clair des modifications apportées au code et permettent de résoudre les conflits potentiels lors de la fusion des branches. Les merge requests sont donc un outil précieux pour assurer la qualité du code et la coordination dans les projets de développement logiciel.

Refactor

Le refactoring est le processus d'amélioration du code existant sans en modifier le comportement externe. Il vise à rendre le code plus lisible, maintenable et performant. Cela implique des actions telles que l'élimination des duplications, la simplification des algorithmes et l'application de bonnes pratiques de codage. Le refactoring est une pratique essentielle pour maintenir un code de qualité et faciliter son évolution.

T D D

Le TDD (Test-Driven Development), ou développement piloté par les tests, est une approche de développement logiciel dans laquelle les tests unitaires sont écrits avant même d'écrire le code de production. Le processus TDD suit généralement trois étapes : écrire un test unitaire qui échoue, écrire le code de production pour faire passer le test et enfin refactoriser le code pour l'améliorer tout en conservant les tests fonctionnels.

Mock

Le mocking consiste à créer des objets factices qui imitent le comportement des objets réels dans le système. Ces objets fictifs, appelés "mocks", permettent de simuler les interactions entre les différentes parties du système et de vérifier les appels aux méthodes et les retours de valeur.

Stubs

Le stubbing, quant à lui, est une technique similaire mais se concentre sur le remplacement de certaines parties du système par des objets factices appelés "stubs". Les stubs sont utilisés pour fournir des réponses prédéfinies lorsqu'une méthode est appelée, sans simuler complètement le comportement du véritable objet.
